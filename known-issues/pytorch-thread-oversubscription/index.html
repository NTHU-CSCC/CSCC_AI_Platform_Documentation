    PyTorch / NumPy Thread Oversubscription in IDE Pods¶

    Status: Known issue · Platform-level fix proposed First observed: 2026-05-13 Affected component: IDE workspace pods (jupyter / vscode) running on GPU nodes Severity: High — silently turns GPU jobs into CPU-bound jobs for students

    1. Discovery / Symptoms¶

    Reproduction case¶

    • Pod: ide-u26000007-jupyter in proj-p26000116-test, scheduled on gpu3
    • GPU allocation: RTX 5090 × sm=25% via NVIDIA DRA + MPS (template gpu-rtx5090-cnt1-sm25-memmax)
    • Pod CPU limit: 16 cores, memory 8Gi
    • Workload: python3 pruning.py (HW4-Network_Compression, AlexNet on CIFAR-10)

    Observed metrics¶

    nvitop showed the training process:

    Metric Value
    GPU SM utilization 0–3 %
    GPU VRAM (process) 694 MiB
    CPU utilization 1642 % (≈ 16.4 cores, i.e. cgroup quota fully saturated)
    Host RAM (process) 1528 MiB
    Threads 260

    The process was on GPU (CUDA context allocated, 694 MiB VRAM held), but the GPU was virtually idle while the CPU quota was pinned at 100 % wall-clock for the entire training run.

    pruning.py itself is well-written: it picks cuda if available and calls .to(device) on the model and every batch. Code is not the problem.

    2. Root Cause¶

    PyTorch (via MKL / OpenMP) reads CPU count from /proc/cpuinfo / os.cpu_count(), neither of which respects cgroup v2 CPU quotas. Inside the pod:

    $ nproc
384                                  # ← host's full CPU count, NOT the pod's 16-core limit

$ cat /sys/fs/cgroup/cpu.max
1600000 100000                       # ← actual quota: 16 cores

>>> torch.get_num_threads()
192                                  # ← PyTorch picks ~nproc/2 by default
>>> torch.get_num_interop_threads()
192

    The result:

    1. PyTorch launches ~192 intra-op + 192 interop OMP/MKL threads.
    2. The kernel cgroup throttles all those threads into 16 cores of actual CPU time.
    3. → Massive context switching, OMP barrier waits, and lock contention.
    4. → 1642 % CPU is almost all wasted work (threads spinning at barriers, not computing).
    5. → CPU-side ops (DataLoader transforms, torch.where masking, kernel launches) become orders of magnitude slower than they should be.
    6. → GPU idles waiting for the next batch.

    This is compounded by the workload being inherently small (AlexNet + 32×32 CIFAR-10 + batch_size=50 + num_workers=0), which already makes the GPU CPU-bound. Thread oversubscription turns "underutilized" into "essentially not used at all."

    Why it's nasty¶

    • It is silent: torch.cuda.is_available() returns True, the model is on GPU, training "works."
    • It only manifests as slow training — students will assume the platform is slow, not that their threading defaults are wrong.
    • It is invisible to anyone who isn't profiling threads and cgroups together.

    Confirmation¶

    The user manually applied either of these and saw a "大幅減少" of training time:

    • In code: torch.set_num_threads(4); torch.set_num_interop_threads(2)
    • In env: OMP_NUM_THREADS=4 MKL_NUM_THREADS=4 python3 pruning.py

    Both work. Both require the student to know about the issue.

    3. Fix Options¶

    3.1 User-side workaround (immediate, no platform change)¶

    Either:

    # Top of script, before torch ops
import torch
torch.set_num_threads(4)
torch.set_num_interop_threads(2)

    Or:

    OMP_NUM_THREADS=4 MKL_NUM_THREADS=4 python3 pruning.py

    Pick ~25–50 % of the pod's CPU limit, not more. For a 16-core pod, 4–8 threads is typical.

    Cons: every student needs to know about this. Not scalable.

    3.2 Platform-side fix (proposed) — auto-inject env vars¶

    Inject thread-count env vars into every IDE container, derived from the container's CPU limit via the downward API. Students get correct defaults without changing a single line of code.

    # In the IDE container spec, generated by the backend
env:
  - name: OMP_NUM_THREADS
    valueFrom:
      resourceFieldRef:
        containerName: ide
        resource: limits.cpu
        divisor: "1"
  - name: MKL_NUM_THREADS
    valueFrom:
      resourceFieldRef:
        containerName: ide
        resource: limits.cpu
        divisor: "1"
  - name: NUMEXPR_NUM_THREADS
    valueFrom:
      resourceFieldRef:
        containerName: ide
        resource: limits.cpu
        divisor: "1"
  - name: OPENBLAS_NUM_THREADS
    valueFrom:
      resourceFieldRef:
        containerName: ide
        resource: limits.cpu
        divisor: "1"

    The resourceFieldRef downward API rounds CPU limits up to the next integer (e.g. 0.5 → 1, 16 → 16, 15.5 → 16). For our IDE pods all CPU limits are already integers, so this is a clean mapping.

    PyTorch, NumPy, NumExpr, OpenBLAS, and any MKL-linked library all respect these env vars at process start. No code change in user scripts.

    Implementation point in the codebase: CSCC_AI_Platform_Backend/internal/application/ide/ide_config.go — function ideContainerEnv(img PredefinedImage, podName, username string) []corev1.EnvVar already builds the env-var list for the IDE container. The thread-count vars should be appended there so they apply to every IDE pod regardless of image.

    3.3 Alternative / complementary fixes (rejected for now)¶

    Option Why not
    Kyverno mutation rule Doable, but keeping all IDE-container env in one Go function is easier to test and ship.
    Set thread count from the IDE image's entrypoint Image-specific; we'd have to patch every IDE image. Env vars in the pod spec are image-agnostic.
    Hard-code OMP_NUM_THREADS=4 Doesn't scale with the pod's actual CPU limit. A 64-core pod would be silently capped at 4.
    Pass through CUDA-side fix only Only helps PyTorch; NumPy / sklearn / xgboost users still suffer.

    4. Validation Plan¶

    After the platform fix lands:

    1. Launch a fresh IDE pod with cpu: 16. Inside the pod: 
      env | grep -E 'OMP|MKL|OPENBLAS|NUMEXPR'
# Expect each one to print 16
python3 -c 'import torch; print(torch.get_num_threads())'
# Expect 16
    2. Re-run pruning.py unchanged (no torch.set_num_threads in the script). Compare wall-clock with the same image before the fix.
    3. Verify nvitop shows CPU close to the actual useful work (single-digit cores worth of activity) and GPU SM% trending upward.
    4. Spot-check a cpu: 4 pod and a cpu: 32 pod to confirm the value scales.
