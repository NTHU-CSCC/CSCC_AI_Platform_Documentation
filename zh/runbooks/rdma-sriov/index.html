    Runbook: SR-IOV RDMA (GPUDirect) on the 25G Mellanox fabric¶

    Add RDMA acceleration for GPU workloads as an additive Multus secondary network using SR-IOV VFs on the 25G Mellanox PFs, via the NVIDIA + SR-IOV Network Operators. The Cilium primary CNI and the 192.168.109.x access lifeline are untouched. Driver: scripts/setup/16-install-rdma-sriov.sh.

    Run only after the Cilium migration (15-) is stable and soaked.

    The fabric & the guardrails¶

    Fabric IPs NIC Role
    25G storage 192.168.110.1-3, pool .220-239 enp193s0f0np0 (gpu1/3), enp194s0f0np0 (gpu2) Longhorn macvlan lh25g0 + RDMA VFs added here
    10G cluster 192.168.110.10X 10G PF Cilium native routing (do not touch)
    109.x access 192.168.109.x separate mgmt NIC ingress/OpenVPN/SSH — never island
    • SR-IOV VFs coexist with the PF IP and the lh25g0 macvlan. Only firmware/driver/switchdev transitions bounce the PF link — hence every change is rolling, per drained node, with a reboot.
    • After each node: lh25g0 storage macvlan back + ingress 192.168.109.201 reachable (the verify-node gate enforces both).
    • R0 must prove 192.168.109.x lives on a different NIC than the 25G PF before anything changes.

    Procedure¶

    cd /home/user/CSCC_AI_Platform

# R0 — read-only hardware review. GATES everything. Inspect every node:
#   Mellanox NICs, rdma link, firmware (mlxconfig SRIOV_EN/NUM_OF_VFS), current VFs,
#   IOMMU cmdline, nvidia-peermem, and CONFIRM 109.x is a separate NIC from the 25G PF.
scripts/setup/16-install-rdma-sriov.sh review
#   -> Decide RDMA_VF_COUNT (default 8). Do NOT proceed if 109.x shares the 25G NIC.

# R1 — per-node prereqs: IOMMU kernel args + mlxconfig SRIOV_EN=1 NUM_OF_VFS=N. Reboots the node.
RDMA_CONFIRM=prereqs-gpu3 scripts/setup/16-install-rdma-sriov.sh prereqs gpu3
RDMA_CONFIRM=prereqs-gpu2 scripts/setup/16-install-rdma-sriov.sh prereqs gpu2
RDMA_CONFIRM=prereqs-gpu1 scripts/setup/16-install-rdma-sriov.sh prereqs gpu1
#   Each run gates on verify-node (storage macvlan + ingress back).

# R2 — operators (NVIDIA Network Operator bundles SR-IOV operator, RDMA CNI, device plugin,
#       MOFED + nvidia-peermem for GPUDirect).
scripts/setup/16-install-rdma-sriov.sh operators

# R3 — VF policy on the 25G PFs. The SR-IOV operator drains+reconfigures node-by-node;
#       gate each node:
scripts/setup/16-install-rdma-sriov.sh vf-policy
scripts/setup/16-install-rdma-sriov.sh verify-node gpu3
scripts/setup/16-install-rdma-sriov.sh verify-node gpu2
scripts/setup/16-install-rdma-sriov.sh verify-node gpu1

# R4 — expose the VF pool as a Multus secondary network (auto-generates a NAD).
scripts/setup/16-install-rdma-sriov.sh network

    Workload usage¶

    A GPU pod requests a VF + the secondary network:

    metadata:
  annotations:
    k8s.v1.cni.cncf.io/networks: default/rdma-vf25g
spec:
  containers:
  - resources:
      limits:
        rdma_vf25g: "1"        # advertised as <prefix>/rdma_vf25g by the SR-IOV device plugin
        nvidia.com/gpu: "1"

    This coexists with the Cilium primary network and the existing cscc-storage storage NAD.

    Validation¶

    • rdma/... / VF resources advertised: kubectl describe node gpu1 | grep -i rdma.
    • ib_write_bw / ib_read_bw between two GPU nodes over the VFs.
    • A 2-node NCCL all-reduce shows GPUDirect RDMA bandwidth (not TCP fallback).
    • Storage intact on all nodes: lh25g0 present, Longhorn RWX volumes mount.

    Rollback¶

    SR-IOV is additive. To revert: delete SriovNetwork/SriovNetworkNodePolicy, uninstall the operators, then per drained node mlxconfig -d <pci> set SRIOV_EN=0 and reboot. The Cilium primary network and the lh25g0 storage macvlan remain intact throughout.

    Key config (overridable env; defaults in scripts/setup/lib/common.sh + this script)¶

    Var Default Meaning
    RDMA_VF_COUNT 8 VFs per 25G PF
    RDMA_RESOURCE_NAME rdma_vf25g SR-IOV resource name
    RDMA_NODE_IFACES gpu1=enp193s0f0np0 gpu2=enp194s0f0np0 gpu3=enp193s0f0np0 25G PF per node
    RDMA_NAD_NAME / RDMA_NAD_NAMESPACE rdma-vf25g / default generated NAD
